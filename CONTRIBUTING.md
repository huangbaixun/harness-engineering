# Contributing to Harness Engineering Plugin

Thank you for your interest in this project! This document explains how to contribute.

## Types of Contributions

This project accepts three types of contributions, each with different requirements and processes:

### 1. Adding or Improving Skills

Skills are the most welcome form of contribution. **All Skill additions and modifications must go through the skill-creator workflow** — this is an official Anthropic meta-skill that provides a structured process of drafting, testing, evaluating, and iterating, ensuring every Skill meets an objective quality baseline.

**Standard Process (skill-creator workflow):**

**Step 1 — Draft SKILL.md**

Create a Skill draft in `skills/your-skill-name/SKILL.md` with YAML frontmatter:

```markdown
---
name: your-skill-name
description: >
  Trigger condition description. Activates when the user mentions...
  Even if the user only mentions..., this Skill should be used.
---
```

**Step 2 — Write eval test cases**

Write tests in `skills/your-skill-name/evals/evals.json` in a skill-creator compatible format:

```json
{
  "skill_name": "your-skill-name",
  "evals": [
    {
      "id": 1,
      "eval_name": "basic-functional",
      "type": "functional",
      "prompt": "Typical user input that triggers this Skill",
      "expected_output": "Description of what the Agent should do",
      "files": [],
      "assertions": [
        { "name": "Checkpoint description", "check": "Objectively verifiable criteria" }
      ]
    },
    {
      "id": 2,
      "eval_name": "constraint-pressure",
      "type": "pressure",
      "rule_under_test": "A key constraint from the Skill",
      "prompt": "User input pressuring the Agent to skip the constraint",
      "expected_output": "Description of how the Agent should uphold the constraint",
      "files": [],
      "assertions": [
        { "name": "Constraint upheld", "check": "Output does not contain any compromise" }
      ]
    }
  ]
}
```

**Step 3 — Run skill-creator eval**

Use skill-creator to launch the eval runner, which runs both a with-skill group and a baseline (no Skill) group simultaneously, then generates a comparison report:

```
In Claude Code, trigger skill-creator and say:
"Please run eval on skills/your-skill-name,
  the eval file is at skills/your-skill-name/evals/evals.json"
```

skill-creator will automatically generate an eval-viewer HTML page for you to review the output differences and assertion pass rates between with-skill and baseline.

**Step 4 — Iterate based on results**

Leave feedback in the eval-viewer. skill-creator will revise SKILL.md accordingly and start the next iteration, repeating until all functional evals pass and pressure eval constraints remain unbroken.

**Step 5 — Description optimization (optional but recommended)**

Use skill-creator's description optimization feature (`run_loop.py`) to fine-tune the `description` field for trigger accuracy.

**Skill directory structure:**

```
skills/
└── your-skill-name/
    ├── SKILL.md                    # Required
    └── evals/
        └── evals.json              # Required, skill-creator compatible format
```

**PR requirements**: Include a benchmark screenshot or `benchmark.json` summary generated by skill-creator, showing the assertion pass rate comparison between with-skill and baseline.

### 2. Adding Language Templates

Language templates live in `docs/templates/<language>/` and help engineers quickly initialize an `AGENTS.md` for a given language stack.

**File structure:**

```
docs/templates/<language>/
├── AGENTS.md.template    # Required — includes testing frameworks, lint tools, and architectural conventions for the language
└── README.md             # Optional — describes the template's intended use cases and key conventions
```

> Note: Older templates used the name `CLAUDE.md.template`. New templates should use `AGENTS.md.template` (cross-tool compatible).

**Template content requirements:**
- Specify test frameworks and commonly used assertion libraries (e.g., JUnit 5 + AssertJ)
- Specify lint / formatting tools (e.g., Checkstyle + SpotBugs)
- List project architecture layers (e.g., entity -> repository -> service -> controller)
- List language-specific prohibitions (e.g., raw types, silent `catch Exception`)
- Use `.claude/` as the standard configuration directory with no hardcoded paths (refer to `docs/templates/generic/AGENTS.md.template`)

See existing templates for reference: `docs/templates/generic/AGENTS.md.template`

---

### 3. Improving Hook Scripts

Hook scripts live in `scripts/` and correspond to events registered in `hooks/hooks.json`.

**Conventions:**
- Scripts must be **silent on success** (no output) and only print error messages on failure
- Use `${CLAUDE_PLUGIN_ROOT}` to reference paths within the plugin — do not use absolute paths
- Each script should do one thing; avoid combining multiple responsibilities
- If modifying an existing script, explain in your PR why creating a new script was not appropriate

---

## PR Guidelines

**Title format:** `type(scope): description`

- `feat(skill)`: New Skill
- `feat(template)`: New language template
- `fix(hook)`: Hook script fix
- `fix(agent)`: Agent behavior fix
- `docs`: Documentation improvement
- `eval`: New or improved eval cases

**PR description must include:**

```markdown
## Changes
<!-- What was done -->

## Trigger Conditions (Skill PRs only)
<!-- Which user inputs trigger this Skill -->

## Eval Verification
<!-- Include the eval case IDs you wrote and describe expected behavior -->

## Manual Testing
<!-- Screenshots or output excerpts from testing with Claude Code -->
```

---

## Local Verification

```bash
# Extract the latest .skill package to a test directory
unzip harness-engineering.skill -d /tmp/harness-test

# Load the plugin for local verification
claude --plugin-dir /tmp/harness-test
```

To verify the plugin loaded successfully, start Claude Code and type "Help me initialize Harness for this project" in the conversation. This should automatically trigger the `harness:init` Skill.

> When adding new Skills or Hook scripts, use `.claude/` as the standard configuration directory.

---

## Code of Conduct

This project follows the [Contributor Covenant](https://www.contributor-covenant.org/) Code of Conduct.

Core principle: We welcome contributors of all backgrounds and experience levels. If your PR is declined, maintainers will explain why and offer suggestions for improvement whenever possible.
